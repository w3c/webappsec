### [Web Crypto](https://github.com/w3c/webcrypto/)

Daniel: It's been a while since we discussed Web Crypto. Biggest update is the modern algorthms draft: PQ algorithms and some others specified either by NIST or CFRG. We have some implementation work going on now: Chrome has implemented ML-KEM and ML-DSA and ChaChaPoly behind a feature flag. Firefox is working on ML-KEM, ML-DSA, ChaChaPoly, and SHA3. Safari's expressed interest in the same algorithms, without ChaChaPoly. Implementations also in Ladybird, Servo, and NodeJS. Also have others like Argon2, etc.

...: Would be nice if we could get to common agreement around which algorithms to implement. Browsers have similar but not quite the same subsets. So we've added adoption guidance to the draft based on our opinion of which algorithms are most urgently needed, but up for debate. It'd be great if folks could take a look at that section of the draft. It would be nice to land on the union of all of these, rather than the intersection! https://wicg.github.io/webcrypto-modern-algos/#adoption-guidance

...: Chrome looking at a hybrid ML-KEM/X25519 implementation. Original idea was to specify all the parts needed for applications to combine into a hybrid KEM, as different applications have landed on different ways to combine them. Having the individual pieces is nice. Chrome's concerned about performance: exposing a single operation is better than combining pieces. Also some concern about exposing SHA3, which is necessary in many constructions. I'd personally prefer to get SHA3 exposed as well, but that depends on crypto libraries and so on.

Mike: Different feedback from different vendors. What are you looking for?

Daniel: Bad if developers can't use algorithms across browsers. In the adoption guidance, we give use cases for the algorithms, why they're necessary. Hopefully we can eventually implement these algorithms after all. If concerns can't be overcome, perhaps we should drop some.

Mike: Perhaps we could pull some algorithms with more agreement into Web Crypto, and leave the rest in the draft to solidify support?

Daniel: Might be reasonable for ML-KEM and ML-DSA.

...: There's a list of other topics that various browsers expressed interest in. Chrome's interested in a synchronous API for performance. Brave has expressed interest in specifying hardware-based crypto. Firefox (and Chrome?) is interested in HPKE. Older proposal from Chrome and others to add streaming. For all these the story is the same: some parties are interested, others not so much. Not clear what to prioritize or specify.

...: Session at the Web Engine Hackfest next week to discuss Web Crypto at greater length. Would be nice to discuss with you more there.

...: Finally: the WG charter. Simone added some text about potentially adding new features to Web Crypto, rather than the current text that only supports "maintenance". Also to clarify the collaboration with IETF.
